## Context

The current system is built on a monolithic SQLite database managed by `aiosqlite`. While functional for a single user, it lacks the scalability, structured schema-less flexibility, and production-grade reliability required for an AI-powered ITR filing platform. Additionally, the existing React frontend features a 445-line document extraction component (`IdentityVerification.jsx`) that is deeply coupled with a specific batch processing flow, leading to high maintenance overhead and a cluttered user experience.

## Goals / Non-Goals

### Goals:
- Migrate the entire persistence layer to MongoDB using the `motor` async driver.
- Implement a pure async/await backend architecture (FastAPI -> Motor).
- Provide a high-fidelity, minimalist authentication UI (Login/Signup).
- Clear out all legacy code related to the previous SQLite schema (taxpayers table).
- Ensure the administrative user profile is correctly seeded with a deterministic SHA-256 OID.

### Non-Goals:
- Implementing secure JWT-based session management (for now, simple response validation is sufficient).
- Redesigning the document extraction engine itself (only the persistence layer of that engine is updated).
- Adding complex multi-step filing workflows beyond authentication.

## Decisions

### 1. Choice of MongoDB Driver: `motor`
We will use `motor` (the official async wrapper for `pymongo`). It integrates perfectly with FastAPI's `asyncio` event loop and allows for non-blocking database operations.

### 2. User Identification: Deterministic OID
Instead of MongoDB's default `ObjectId`, we will use a custom `oid` generated by hashing the user's core identity fields (Name, PAN, Aadhar, Mobile, Email). This ensures data integrity and prevents duplicate profiles even if the database is restored from different backups.

### 3. Timezone Handling: IST (UTC+5:30)
All `created_at` and `updated_at` timestamps will be stored as IST-aware datetime objects to ensure consistency with Indian tax filing deadlines and local user expectations.

### 4. UI Library: Pure CSS + React
We will continue using standard CSS with a dark-mode theme to keep the bundle size small and the design pixel-perfect. Lucide-React icons will be used for premium iconography.

## Risks / Trade-offs

- **Risk**: MongoDB connection string exposure.
- **Mitigation**: Strictly enforced `.env` usage and `Settings` singleton.
- **Trade-off**: By simplifying the UI to only AuthPage, we temporarily hide the document extraction features. 
- **Rationale**: The extraction flow was deemed too "noisy" for the initial landing experience. Keeping the APIs alive ensures feature-parity for future restoration.
- **Trade-off**: Using a string-based OID instead of `ObjectId` might have marginal performance impacts on very large collections.
- **Rationale**: The benefit of deterministic mapping for identity verification outweighs the negligible lookup delta in this domain.
